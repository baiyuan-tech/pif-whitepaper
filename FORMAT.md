# Whitepaper Format Specification / 白皮書格式規範

*This document defines the authoring, linting, and PDF-build conventions for the PIF AI Whitepaper. Heavily inspired by (and structurally compatible with) the [`baiyuan-tech/geo-whitepaper`](https://github.com/baiyuan-tech/geo-whitepaper) sibling project.*

本文件定義 PIF AI 白皮書的撰寫、linting 與 PDF 建置慣例，結構與姊妹專案 [`baiyuan-tech/geo-whitepaper`](https://github.com/baiyuan-tech/geo-whitepaper) 相容。

---

## 1. 目錄結構 / Directory Layout

```
docs/whitepaper/
├── README.md                  # Top-level entrypoint (bilingual)
├── FORMAT.md                  # This file
├── whitepaper.md              # (Generated) single-file concatenation for LLM consumption
├── whitepaper.pdf             # (Generated) compiled PDF
├── .markdownlint.jsonc
├── assets/
│   ├── figures/               # Mermaid / SVG sources
│   └── pdf/
│       ├── concat.sh          # Chapter concatenation for Pandoc
│       ├── metadata-zh-TW.yaml
│       └── metadata-en.yaml
├── zh-TW/
│   ├── README.md
│   ├── ch01-abstract.md       # One chapter per file
│   ├── ch02-regulatory-background.md
│   ├── ... (ch01 – ch12)
│   ├── appendix-a-glossary.md
│   ├── appendix-b-api-endpoints.md
│   ├── appendix-c-references.md
│   └── appendix-d-changelog.md
└── en/
    └── (mirror of zh-TW)
```

Rules:

- **One H1 per file.** No more than one `#`-level heading per chapter file.
- **Per-chapter filenames** follow `chNN-kebab-slug.md` (NN = 01..12).
- **Appendix filenames** follow `appendix-<letter>-<slug>.md`.
- The consolidated `whitepaper.md` / `.pdf` files are generated artifacts; they are checked in at release time only.

## 2. YAML Frontmatter / 章節前置資料

Every chapter file begins with YAML frontmatter:

```yaml
---
title: "第 N 章：章節標題"
description: "一行中文摘要（120 字以內）"
chapter: N
lang: zh-TW
authors:
  - "Vincent Lin"
keywords:
  - "關鍵詞1"
  - "關鍵詞2"
word_count: approx
last_updated: 2026-04-19
---
```

English files use `lang: en` and English metadata.

## 3. Heading Hierarchy / 標題層級

- `#` — Chapter title (only one per file)
- `##` — Main sections, numbered `N.1`, `N.2`, …
- `###` — Sub-sections, numbered `N.1.1`, `N.1.2`, …
- `####` — Avoid unless necessary

Each `##` heading anchors are auto-generated by GitHub; cross-chapter links use relative paths + anchor.

## 4. Chapter Anatomy / 章節結構

Each chapter MUST contain, in order:

1. YAML frontmatter (§2)
2. `# Chapter title`
3. Blockquote introduction (2-3 sentences summarizing the chapter)
4. `## N.1 …` sections
5. `## Key Takeaways / 重點` — 3-5 bullet points
6. `## References / 參考資料` — Numbered list
7. `## Revision history / 修訂記錄` — Table
8. Navigation footer: `**導覽** prev | up | next`

## 5. Diagrams / 圖表

**Prefer Mermaid** for GitHub-native rendering. PNG/SVG allowed for what Mermaid cannot express.

- Node IDs **MUST be ASCII** (labels may contain CJK)
- Labels ≤ 12 characters (bilingual format: `中文 / English`)
- Every figure is captioned: `**圖 N.M 說明**：...`
- Mermaid sources are also stored in `assets/figures/NN-slug.mmd` for editing

## 6. Code Blocks / 程式碼

Always specify a language fence (`python`, `typescript`, `sql`, `bash`, `yaml`, `mermaid`, `text`).

Inline code references source files with the pattern `file/path.py:LINE` (e.g., `app/services/rag_client.py:128`) so readers can jump to the exact location.

## 7. Tables / 表格

Use GitHub Flavored Markdown pipe syntax. Always provide a header row with alignment markers.

## 8. Privacy & Data Hygiene / 隱私與資料衛生

- Real customer names → anonymize as "Brand A", "Brand B"
- Real CAS numbers / formula concentrations → use representative generic examples
- Real user counts / revenue → use documented targets (`CLAUDE.md` goal) or ranges, never invent numbers
- Never commit secrets — see repository [SECURITY](../../SECURITY.md) and [CONTRIBUTING](../../CONTRIBUTING.md)

## 9. Language & Style / 語言與文體

- ASCII quotation marks (`"…"` not `“…”`) in markdown source
- Space between CJK and Latin: `PIF 建檔` not `PIF建檔`
- ISO 8601 dates: `2026-04-19` not `Apr 19, 2026`
- `**bold**` for defined terms on first use; `_italics_` for emphasis; `` `code` `` for technical identifiers
- Tone: academic-technical; avoid marketing superlatives; cite sources

## 10. AI-Friendly Structured Data / AI 友善結構化資料

Each language-level `README.md` embeds Schema.org `TechArticle` JSON-LD inside an HTML comment:

```html
<!-- AI-friendly structured data
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "TechArticle",
  "headline": "…",
  "author": {"@type": "Person", "name": "Vincent Lin"},
  "publisher": {"@type": "Organization", "name": "Baiyuan Tech"},
  "inLanguage": "zh-TW",
  "datePublished": "2026-04-19",
  "dateModified": "2026-04-19",
  "keywords": "PIF, cosmetic, AI, multi-tenant SaaS"
}
</script>
-->
```

GitHub strips HTML comments so human readers don't see it; AI crawlers that fetch the raw source still get structured metadata.

## 11. PDF Build / PDF 建置

The `.github/workflows/build-pdf.yml` workflow:

1. Installs Pandoc + XeLaTeX + Noto CJK fonts
2. Runs `assets/pdf/concat.sh <lang>` to concatenate chapters → `whitepaper.md`
3. Preprocesses (strip mermaid, emoji, unicode variation selectors — XeLaTeX-unsafe)
4. Runs Pandoc with `metadata-<lang>.yaml` → `whitepaper-<lang>.pdf`
5. Uploads as workflow artifact; on release, attaches to the GitHub Release

Local PDF build:

```bash
bash docs/whitepaper/assets/pdf/concat.sh zh-TW > whitepaper-zh-TW.md
pandoc whitepaper-zh-TW.md \
  --pdf-engine=xelatex \
  --metadata-file=docs/whitepaper/assets/pdf/metadata-zh-TW.yaml \
  -o whitepaper-zh-TW.pdf
```

## 12. Markdown Linting / Markdown Lint

`.markdownlint.jsonc` enforces:

- Consistent list indent (2 spaces)
- Single H1 per file
- ATX-style headings
- Line length: soft limit 120; hard limit 200
- No trailing whitespace
- Code blocks must be fenced (no indent style)

Run `markdownlint docs/whitepaper/` before PR; CI runs it on every push.

## 14. Revision Policy / 修訂政策

- **Minor updates** (typos, link fixes) — direct PR to `master`
- **Content changes** — PR with reviewers from `CODEOWNERS`; changelog entry required in `appendix-d-changelog.md`
- **Major revisions** (new chapter, restructure) — `[RFC]`-prefixed issue first

Versioning follows `vMAJOR.MINOR`:

- MAJOR bumps on content restructure or removal
- MINOR bumps on content addition
